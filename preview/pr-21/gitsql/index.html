gitsql

Point it at any git repository and explore its history, files, authors, and line-by-line blame with plain SQL.

go-git for git access SQLite virtual tables (FDW) pure-Go modernc.org/sqlite — no CGo clones cached locally

gitsql exposes a repository as a handful of SQLite virtual tables — the SQLite equivalent of a foreign data wrapper. Each table is backed by go-git, wired into SQLite through go-sqlite-fdw/modernc, which runs on the pure-Go modernc.org/sqlite engine. Remote repos are cloned once into a local cache and reused on later runs, so re-querying is instant and works offline.

Every result on this page is real output from gitsql BurntSushi/ripgrep (≈2,200 commits since 2016), captured with the bundled example queries.

Quick start

# build it (Go 1.25+; the toolchain auto-downloads)
cd gitsql && go build -o gitsql .

# run an example query against a public repo (clones + caches on first run)
./gitsql --example top-authors BurntSushi/ripgrep

# run your own SQL
./gitsql BurntSushi/ripgrep "SELECT count(*) FROM commits"

# or open an interactive prompt over the current repo
./gitsql .

Flags come before the repo: gitsql [flags] [repo] [sql]. repo is a local path, a clone URL, or an owner/repo GitHub shorthand (default .). Output formats: table (default), csv, tsv, json, md.

The tables

TableOne row per…Notable columns
commitscommit reachable from a ref (HEAD by default) hash, author_name, author_when, author_unix, message, summary, parents, is_merge
refsbranch, tag, remote, or HEAD ref name, short_name, type, target, hash, is_branch, is_tag, is_head
tagstag (annotated or lightweight) name, type, target, tagger_name, tagger_when, message
filesfile in the tree at a ref path, name, mode, type, size, blob_hash, is_binary, lines + hidden contents
commit_filesfile changed by a commit (like --numstat) commit_hash, path, old_path, change, additions, deletions, binary
blameline of a file (needs WHERE path = '…') path, line_no, content, commit_hash, author_name, author_when

Tables commits, files, commit_files, and blame accept a hidden ref column, e.g. … WHERE ref = 'v1.0.0'; files also has a hidden contents column for searching blob text. Timestamps come in two flavors: *_when is the author's wall clock (so strftime('%H', author_when) is their local hour) and *_unix is the true epoch.

Fun queries

Repository at a glance

A one-row summary built from four tables at once.

SELECT
  (SELECT count(*) FROM commits)                            AS commits,
  (SELECT count(*) FROM refs WHERE is_branch)             AS branches,
  (SELECT count(*) FROM tags)                                AS tags,
  (SELECT count(*) FROM files)                               AS files_at_head,
  (SELECT min(date(author_unix, 'unixepoch')) FROM commits) AS first_commit,
  (SELECT max(date(author_unix, 'unixepoch')) FROM commits) AS latest_commit;

What hour do people commit? (author's local clock)

strftime('%H', author_when) over commits. ripgrep is an evening project.

SELECT strftime('%H', author_when) AS hour, count(*) AS commits
FROM commits GROUP BY hour ORDER BY hour;

Net lines per author — a JOIN across two virtual tables

commit_files (numstat) joined to commits on the commit hash. Merge commits are excluded from the scan, so a branch's work isn't double-counted.

SELECT c.author_name,
       sum(cf.additions)                AS added,
       sum(cf.deletions)                AS removed,
       sum(cf.additions - cf.deletions) AS net
FROM commit_files cf
JOIN commits c ON c.hash = cf.commit_hash
WHERE cf.binary = 0
GROUP BY c.author_name ORDER BY added DESC LIMIT 6;

Follow one file's history

Every commit that touched README.md — commit_files joined to commits.

SELECT c.author_when, substr(cf.commit_hash, 1, 8) AS rev,
       cf.change, cf.additions, cf.deletions, c.author_name
FROM commit_files cf JOIN commits c ON c.hash = cf.commit_hash
WHERE cf.path = 'README.md' ORDER BY c.author_unix DESC;

Across repositories — shared commits & ahead/behind

Bind a second repo and compare commit sets. go-git/go-git is the community continuation of the archived src-d/go-git: same 2015 origin, src-d froze in 2020, the fork carried on.

CREATE VIRTUAL TABLE orig USING git_commits('src-d/go-git');

-- commits in both histories
SELECT count(*) AS shared FROM commits c JOIN orig o ON c.hash = o.hash;

-- how far each is ahead of the other
SELECT (SELECT count(*) FROM commits WHERE hash NOT IN (SELECT hash FROM orig))    AS gogit_ahead,
       (SELECT count(*) FROM orig    WHERE hash NOT IN (SELECT hash FROM commits)) AS srcd_only;
shared
------
1538

gogit_ahead  srcd_only
-----------  ---------
2171         2

The two repos share 1,538 commits, the fork is 2,171 ahead, and src-d has just 2 commits the fork never took — its final "this project has moved" README note (and its merge). srcd_only = 0 would mean the fork is a strict fast-forward.

How it works

A repo spec is resolved by a small manager: local paths open in place; remote URLs and owner/repo shorthands are bare-cloned into <cache>/gitsql/repos/… and reused thereafter (--update fetches, --offline forbids the network). Each table is an fdw.Source whose cursor pulls from go-git (a commit-time-ordered log, a recursive tree walk, a numstat diff, a blame), and SQLite's BestIndex hook pushes down equality on keys like hash, commit_hash, and path so point lookups stay fast. Expensive per-commit diffs are memoized for the life of the process.
