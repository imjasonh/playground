# Design: iOS apps with TestFlight CD and PR previews

> Status: **proposal / design only**. Nothing here is wired up yet. This document
> describes how to add first-class iOS apps to the playground so they get the
> same low-ceremony treatment as browser apps: **continuous delivery to
> TestFlight on merge to `main`**, and a **per-PR preview** you can install on a
> device from a link/QR code in a preview tab, mirroring the existing gh-pages
> preview flow.

## 1. Goals & how they map to what we already do

Today a browser app is a top-level directory with `index.html`. On merge to
`main`, `deploy.yml` publishes it to GitHub Pages; on a PR, `preview.yml` copies
it under `preview/pr-<N>/` on the `gh-pages` branch and comments the URL;
`test.yml` runs its tests; `cleanup.yml` removes the preview when the PR closes.
Discovery is per top-level directory, keyed off a marker file (`index.html`,
`go.mod`, `Cargo.toml`).

We want the iOS equivalent:

| Browser today | iOS equivalent (proposed) |
|---------------|---------------------------|
| Marker: `index.html` | Marker: `project.yml` (XcodeGen) at the app dir root |
| `test.yml` runs `npm test` | New macOS test leg runs `xcodebuild test` on a Simulator |
| Merge to `main` â†’ deploy to Pages | Merge to `main` â†’ archive, sign, upload to **TestFlight** |
| PR â†’ live preview URL in a comment | PR â†’ **installable preview** (OTA ad-hoc + QR, or Simulator build artifact) linked from the preview tab and PR comment |
| `cleanup.yml` removes preview dir | Same: remove the PR's preview dir / expire the preview build |

The honest headline up front: **iOS previews cannot be as frictionless as a URL.**
Apple has no concept of an ephemeral, publicly-installable per-PR app. There are
three viable preview mechanisms, each with a real tradeoff (Section 5). CD to
TestFlight on `main`, by contrast, maps cleanly and is the well-trodden path.

## 2. What has to change vs. the browser model

Three things make iOS fundamentally different from the current all-`ubuntu-latest`,
zero-credentials, static-hosting setup:

1. **Build host.** iOS builds require **macOS + Xcode**. That means a
   `macos-latest` GitHub-hosted runner (Xcode preinstalled). Cost note in
   Section 7.
2. **An Apple account + code signing.** TestFlight and on-device installs require
   an **Apple Developer Program** membership, an **App ID / bundle identifier**,
   signing **certificates**, and **provisioning profiles**. None of this is free
   or automatable end-to-end without a human doing first-time setup in Apple's
   portals (Section 8).
3. **Distribution is not just "copy files."** GitHub Pages can host the *install
   assets* for OTA previews (an `.ipa` plus a `manifest.plist`, served over the
   HTTPS that `itms-services://` requires), but the *build itself* must be signed
   and, for TestFlight, uploaded through App Store Connect.

Everything else â€” per-directory discovery, PR comments, preview-dir cleanup â€”
reuses the patterns already in `.github/`.

## 3. Repo conventions for an iOS app

Define a new app type consistent with the existing rules in `AGENTS.md`.

**An iOS app is a non-hidden top-level directory containing `project.yml`**
(an [XcodeGen](https://github.com/yonaskolb/XcodeGen) spec). We key off
`project.yml` rather than an `*.xcodeproj` because an `.xcodeproj` is a bundle
directory full of machine-generated, merge-conflict-prone files; a single
declarative YAML that regenerates the project in CI is far cleaner in git and
matches the "one obvious marker file" style of `go.mod` / `Cargo.toml`.
(Tuist's `Project.swift` is an equally good alternative if we prefer Swift.)

Proposed layout, following `kanoodle/` / `web-push/` as references:

```

â”śâ”€â”€ Info.plist             # or generated by XcodeGen
â”śâ”€â”€ ExportOptions.plist    # app-store / ad-hoc export config (or generated per-lane)
â”śâ”€â”€ fastlane/
â”‚   â”śâ”€â”€ Fastfile           # lanes: test, beta (TestFlight), preview (ad-hoc)
â”‚   â””â”€â”€ Appfile            # bundle id, team id, ASC API key wiring
â”śâ”€â”€ Sources/               # Swift sources
â”śâ”€â”€ Tests/                 # XCTest unit/UI tests
â”śâ”€â”€ Resources/             # assets, icons
â”śâ”€â”€ README.md              # build/test/run instructions
â””â”€â”€ .gitignore             # DerivedData/, *.ipa, *.xcarchive, build/, fastlane report files
```

Rules (to add to `AGENTS.md` once we adopt this):

- Each iOS app is **isolated**: it owns its `project.yml`, `fastlane/`, and
  bundle identifier. No repo-root Xcode workspace, no cross-app imports.
- Do **not** commit `index.html` â€” iOS apps are not Pages apps (their *preview
  install assets* may land on gh-pages, but the app is not a static site).
- Do **not** commit build artifacts: `DerivedData/`, `*.xcarchive`, `*.ipa`,
  `build/`, `fastlane/report.xml`, `fastlane/Preview.html`, etc.
- Pin the toolchain: choose the Xcode version explicitly in CI
  (`maxim-lobanov/setup-xcode`) so local and CI agree, analogous to
  `rust-toolchain.toml`.
- Bundle IDs: use a stable scheme, e.g. `io.github.imjasonh.playground.<app>`.

## 4. Discovery (mirror the Rust/Go scripts)

Add `.github/scripts/discover-ios-apps.sh`, a near-copy of
`discover-rust-apps.sh`, with the marker swapped:

```bash
is_ios_app() {
  local name="$1"
  [[ "$name" == .* ]] && return 1
  [[ -f "$name/project.yml" ]]
}
```

Then extend `discover-changed-apps.sh` to also emit an `ios=` output alongside
`apps` / `modules` / `rust`, using the same `--all` (first push to `main`) and
`--from-changes` (diff) logic already there. No other discovery logic changes.

Document the local commands in `AGENTS.md`:

```bash
bash .github/scripts/discover-ios-apps.sh --all
git diff --name-only origin/main...HEAD | bash .github/scripts/discover-ios-apps.sh --from-changes
```

## 5. PR previews â€” the three options (pick one, or offer a ladder)

This is the crux. Ranked by how close they get to "just tap the link," with
their costs.

### Option A â€” Simulator build artifact (zero credentials, not on-device)

CI builds a **Simulator** `.app` (no signing needed at all), zips it, and
uploads it as a **GitHub Actions artifact** on the PR. A reviewer with a Mac
downloads it and drags it onto a running Simulator (`xcrun simctl install`).

- Pros: no Apple account, no certs, no UDIDs, works for any PR immediately;
  fully automatable today.
- Cons: not a real device; requires a Mac + Simulator; not a tap-to-install link.
- Best as the **always-on baseline**, even if we also do B.

### Option B â€” OTA ad-hoc install hosted on gh-pages (the closest analog to the browser preview tab)

CI builds a **signed ad-hoc `.ipa`**, generates a `manifest.plist` pointing at
it, and publishes both under `gh-pages` at `preview/pr-<N>/<app>/` â€” exactly like
the browser preview flow. The preview tab / PR comment then links to
`itms-services://?action=download-manifest&url=â€¦manifest.plist` plus a **QR
code**, so a tester taps it on their iPhone and installs over the air. This reuses
`peaceiris/actions-gh-pages`, `keep_files: true`, and the existing PR-comment
step almost verbatim, and `cleanup.yml` removes the dir on close.

- Pros: real on-device install from a link/QR in the PR â€” the true "preview tab"
  experience; hosting is free on the Pages site we already run; HTTPS requirement
  is satisfied by gh-pages.
- Cons: ad-hoc provisioning only installs on **devices whose UDID is registered**
  in the provisioning profile (Apple cap: 100 devices/type/year). So every tester
  device must be registered once. This is the irreducible friction and the main
  thing that makes iOS previews not fully self-serve.
- Note: `.ipa` and `.plist` must be served with correct MIME types; gh-pages does
  this fine, and `itms-services` needs only the `.plist` URL to be HTTPS.

### Option C â€” TestFlight per-PR build into an internal testing group

CI uploads **every PR build to TestFlight** under the app's record, tagged in the
build metadata/notes with the PR number, distributed to a dedicated **internal
testing group** (up to 100 App Store Connect users; no Beta App Review needed for
internal testers, so builds appear in minutes).

- Pros: uses the real TestFlight app; no per-device UDID registration; testers
  just open TestFlight.
- Cons: TestFlight has **no per-PR "tab"** â€” all PRs pile into one app's build
  list, distinguished only by build number/notes. Testers must be ASC users you
  invite. Burns a build number and an upload per PR. Less "ephemeral" than a
  preview dir. External testers would additionally hit Beta App Review.

### Recommendation

Ship **A** as the baseline (free, universal) and add **B** as the flagship
"preview tab" experience once the Apple account and a handful of tester UDIDs
exist. Treat **C** as optional for teams that live in TestFlight already. B is
what actually reproduces "preview apps in a preview tab, deployed on PRs."

## 6. Continuous delivery to TestFlight on `main`

This is the clean part. Proposed `.github/workflows/ios-deploy.yml`
(push to `main`):

1. Discover changed iOS apps (Section 4). Skip the job if none.
2. `runs-on: macos-latest`; select Xcode with `maxim-lobanov/setup-xcode`.
3. Install tooling: XcodeGen + fastlane (bundler or Homebrew).
4. `xcodegen generate` to produce the `.xcodeproj` from `project.yml`.
5. **Signing** via fastlane **`match`** (recommended) or App Store Connect
   API-key cloud signing. `match` keeps certs + profiles in a separate private
   git repo, encrypted with a passphrase â€” CI fetches them read-only.
6. **Version/build number**: auto-increment the build number (e.g.
   `agvtool`/`fastlane` using the latest TestFlight build number + 1) so every
   upload is unique.
7. `fastlane gym` (build & sign an App Store `.ipa`) â†’ `fastlane pilot`/`deliver`
   uploads to **TestFlight** via the **App Store Connect API key**.
8. Distribute to an **internal** testing group automatically. External groups
   require **Beta App Review** (first build per version) â€” gate that behind a
   manual step or `workflow_dispatch`.

A minimal `Fastfile` sketch:

```ruby
default_platform(:ios)

platform :ios do
  desc "Run unit tests on a Simulator"
  lane :test do
    run_tests(scheme: "MyIosApp", devices: ["iPhone 15"])
  end

  desc "Build + upload to TestFlight"
  lane :beta do
    setup_ci                       # keychain for CI
    match(type: "appstore", readonly: true)
    latest = latest_testflight_build_number rescue 0
    increment_build_number(build_number: latest + 1)
    build_app(scheme: "MyIosApp")
    upload_to_testflight(
      api_key_path: ENV["ASC_API_KEY_PATH"],
      distribute_external: false,  # internal group; flip on for external + review
      skip_waiting_for_build_processing: true
    )
  end

  desc "Build an ad-hoc IPA + manifest for OTA PR preview"
  lane :preview do
    setup_ci
    match(type: "adhoc", readonly: true)
    build_app(scheme: "MyIosApp", export_method: "ad-hoc")
    # emit manifest.plist next to the .ipa for itms-services OTA install
  end
end
```

## 7. CI cost & runner notes

- **macOS minutes are the cost driver.** On GitHub-hosted runners, macOS minutes
  bill at **10Ă—** the Linux rate on **private** repos. `imjasonh/playground` is
  public, so Actions minutes on standard runners are free, but macOS runners have
  lower concurrency and slower cold starts than the ubuntu jobs we run today.
- Keep the iOS legs **change-gated** (only run when an iOS app dir changed), just
  like the current per-type discovery, so we don't pay macOS time on browser/Go/
  Rust-only PRs.
- Cache DerivedData and SwiftPM/CocoaPods to cut build times.
- The existing `test.yml` stays all-Linux; add iOS as a **separate job/workflow**
  on `macos-latest` rather than bolting it onto the Linux `test` job, so the
  single-runner Linux flow is unchanged.

## 8. What YOU need to do (can't be automated from this repo)

These are the human/Apple-side prerequisites. Roughly in order. **A detailed,
beginner-friendly, click-by-click version of this list lives in
[`ios-testflight-setup.md`](ios-testflight-setup.md)** â€” follow that if you've
never done it before; the summary here is just the map.

1. **Enroll in the Apple Developer Program** ($99/yr) for the account/org that
   will own the apps.
2. **Create the App ID / bundle identifier** (e.g.
   `io.github.imjasonh.playground.<app>`) and a matching **app record in App
   Store Connect** for each iOS app you want on TestFlight.
3. **Create an App Store Connect API key** (Users and Access â†’ Integrations â†’
   App Store Connect API), role **App Manager**. Save the **Issuer ID**, **Key
   ID**, and the **`.p8` private key**.
4. **Set up code signing.** Recommended: create a **private git repo for
   fastlane `match`** and run `fastlane match appstore` / `match adhoc` once
   locally to seed certs + profiles; pick a **match passphrase**. (Alternative:
   API-key cloud signing, no cert repo.)
5. **Add GitHub repository secrets** so CI can sign and upload:
   - `ASC_KEY_ID`, `ASC_ISSUER_ID`, `ASC_API_KEY_P8` (the key contents, base64)
   - `APPLE_TEAM_ID` (your 10-character Team ID)
   - `MATCH_GIT_URL`, `MATCH_PASSWORD`, `MATCH_GIT_BASIC_AUTHORIZATION`
     (base64 `user:token` to read the match repo)
   - (These become Cloud Agent secrets too if you want agents to run the lanes.)
6. **Register tester device UDIDs** in the Developer portal for **Option B**
   (OTA ad-hoc previews), and rebuild the ad-hoc profile via `match` when the
   device list changes. Skip if you only use Option A/C.
7. **Create TestFlight testing groups** and invite testers. For **external**
   testers, complete **Beta App Review** and the **export-compliance /
   encryption** questionnaire (can be answered once via `ITSAppUsesNonExempt
   Encryption` in Info.plist).
8. **Provide app metadata & assets**: app icon, display name, and (for external
   TestFlight) a beta description, contact email, and privacy info.
9. **Confirm the runner/cost choice** (Section 7) â€” public repo keeps macOS
   minutes free; if the repo ever goes private, budget for 10Ă— macOS minutes or
   bring a self-hosted Mac runner.
10. **Pick the preview option(s)** from Section 5 you want me to implement
    (recommend A now, add B once UDIDs exist).

## 9. Suggested rollout / implementation order

Once the Section 8 prerequisites (at least an Apple account + ASC API key +
`match`) exist, the in-repo work is:

1. Add `discover-ios-apps.sh` and extend `discover-changed-apps.sh`
   (`ios=` output). *(No Apple creds needed; safe to land first.)*
2. Add a scaffold iOS app (e.g. `hello-ios/` with `project.yml`, a trivial
   SwiftUI view, one XCTest) as the reference app â€” the iOS analog of `hello/`.
3. Add `ios-test.yml` (or an iOS job): `macos-latest` â†’ XcodeGen â†’ `xcodebuild
   test` on a Simulator. *(No signing needed; can run on any PR immediately.)*
4. Add **Option A** preview: upload the Simulator `.app` as a PR artifact.
5. Add `ios-deploy.yml`: TestFlight CD on `main` via fastlane (needs secrets).
6. Add **Option B** preview: ad-hoc IPA + manifest â†’ gh-pages `preview/pr-<N>/`,
   QR + `itms-services` link in the PR comment; extend `cleanup.yml` (already
   removes `preview/pr-<N>` wholesale, so IPAs are cleaned up for free).
7. Update `AGENTS.md` and `README.md` to document the new "iOS app" type,
   discovery marker, and workflows.

Steps 1â€“4 are pure CI/tooling and can land before any Apple account exists;
5â€“6 unlock once the credentials in Section 8 are in place.
